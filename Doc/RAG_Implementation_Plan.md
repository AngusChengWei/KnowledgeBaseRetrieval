# 基于 Vue3 + Python + DeepSeek API 的本地知识库问答系统（完整方案）

## 一、方案概述

本方案为企业提供一套完全本地化的 RAG（检索增强生成）智能问答系统。所有知识文档和向量库均存储在企业内部服务器，仅调用云端 Embedding API 和 DeepSeek API 进行必要的计算，确保数据保密性与合规性。

## 二、系统架构

系统采用前后端分离架构：

- **前端**：Vue3 框架，提供聊天式交互界面。
- **后端**：Python FastAPI 框架，负责文档处理、向量检索、API 调用。
- **向量数据库**：Chroma（本地持久化），存储文档的向量表示。
- **Embedding 服务**：云端 API（如阿里云通用文本向量、硅基流动 bge-m3），将文本转换为向量。
- **大语言模型服务**：DeepSeek API（通过硅基流动或阿里云百炼等国内渠道），根据检索结果生成最终答案。

## 三、数据流程

### 3.1 知识库构建（离线/定时执行）

1. 管理员将内部文档（PDF、Word、TXT 等）放入指定目录。
2. 后端程序扫描目录，解析文档内容。
3. 将长文档切分为固定大小的文本块（每块约 500 字符，相邻块重叠 50 字符）。
4. 对每个文本块调用云端 Embedding API，生成向量。
5. 将向量及原文存入本地 Chroma 数据库。

### 3.2 用户问答（在线实时）

1. 用户在前端输入问题，点击发送。
2. 前端将问题发送到后端 `/ask` 接口。
3. 后端调用 Embedding API 将问题转换为向量。
4. 在 Chroma 中检索与问题向量最相似的 Top‑K 个文本块（K 默认 3）。
5. 若检索到的最高相似度低于阈值（默认 0.65），则直接返回“无法回答”。
6. 否则，将检索到的文本块作为“参考资料”，与用户问题一起组装成提示词。
7. 后端调用 DeepSeek API（temperature=0.1，低随机性）生成回答。
8. 后端将回答及引用来源返回前端展示。

### 3.3 知识库更新

- 用户点击前端“更新知识库”按钮，后端重新执行知识库构建流程（清空旧数据，重新生成所有向量）。

## 四、技术选型

| 组件 | 选择 | 说明 |
|------|------|------|
| 前端框架 | Vue3 + Axios | 轻量、易用，支持流式渲染 |
| 后端框架 | FastAPI | 异步高性能，天然支持 SSE |
| 大语言模型 API | DeepSeek（通过硅基流动/阿里云百炼） | 国内稳定访问，价格低，效果好 |
| Embedding API | 阿里云通用文本向量 / 硅基流动 bge-m3 | 国内服务稳定，无需本地 GPU |
| 向量数据库 | Chroma | 轻量级，持久化，与 Python 集成良好 |
| 文档解析 | PyPDF2、python-docx、chardet | 支持主流办公格式 |
| 部署方式 | 内网服务器（Windows/Linux） | 数据不出企业网络 |

## 五、实施步骤

### 阶段一：环境准备（1-2 天）

- 准备一台内网服务器（推荐 4 核 8G 内存，无需 GPU）。
- 安装 Python 3.10+ 环境。
- 注册硅基流动或阿里云百炼账号，获取 DeepSeek API Key 和 Embedding API Key。
- 在后端项目根目录创建 .env 文件，写入 API 密钥和配置参数。

### 阶段二：后端开发（3-4 天）

- 实现文档解析模块：支持 PDF、DOCX、TXT 格式。
- 实现文本分块逻辑：固定大小分块，带重叠。
- 实现 Embedding API 调用封装（异步请求，批量处理）。
- 实现 Chroma 向量存储与检索（建集合、添加向量、查询、相似度过滤）。
- 实现 DeepSeek API 调用封装（组装提示词、低 temperature 参数）。
- 实现 FastAPI 接口：/ask、/rebuild、/health。

### 阶段三：前端开发（2-3 天）

- 创建 Vue3 项目，安装 Axios。
- 设计聊天界面：消息列表、输入框、发送按钮、加载动画。
- 实现调用 /ask 接口的方法，将响应中的答案和来源渲染到界面。
- 添加“更新知识库”按钮，调用 /rebuild 接口。
- 实现滚动到最新消息、Markdown 简单渲染（换行、加粗）。

### 阶段四：集成测试与部署（1-2 天）

- 将后端部署到内网服务器，使用 uvicorn 运行（建议搭配 Supervisor 或 systemd 守护）。
- 前端构建为静态文件，用 Nginx 托管（配置反向代理解决跨域）。
- 上传真实文档（如员工手册、规章制度）进行功能验证。
- 测试边界问题（超出知识库范围、相似度不足等情况），确认系统能够正确拒绝回答。
- 内部体验并收集反馈。

## 六、关键配置参数

| 参数 | 推荐值 | 作用 |
|------|--------|------|
| 分块大小（chunk_size） | 500 字符 | 平衡语义完整性与检索颗粒度 |
| 重叠（overlap） | 50 字符 | 保持相邻块的上下文连贯 |
| 检索数量（Top‑K） | 3 | 送入大模型的片段数量 |
| 相似度阈值 | 0.65 | 低于此值的片段不送入大模型 |
| temperature | 0.1 | 降低模型创造性，避免编造 |
| max_tokens | 2048 | 限制输出长度 |

## 七、数据安全与边界控制

- **数据本地化**：原始文档、分块文本、向量数据库均存储在企业内网服务器，不离开可控环境。
- **最小外传**：仅将检索到的少量相关片段（通常不足 1500 字符）和用户问题发送给 DeepSeek API，不发送完整文档。
- **回答边界**：
  - 检索相似度低于阈值 → 直接返回“无法回答”。
  - 提示词强制要求模型“不使用外部知识，不编造”。
  - API 参数 temperature 设为 0.1，减少幻觉。
- **审计能力**：系统可记录每次问答的输入、检索到的片段、模型输出，便于事后审计。

## 九、后续演进方向

- **流式输出**：后端改为 SSE 响应，前端逐字渲染，提升体验。
- **混合检索**：增加 BM25 关键词检索，与向量检索融合（RRF 重排），提高召回率。
- **多租户隔离**：为不同部门建立独立的知识库和权限体系。
- **替换后端语言**：当需要更高并发和稳定性时，可将 Python 后端替换为 .NET 8 并提供相同 API。
- **引入 Agent**：当需要多步推理或调用企业业务系统时，升级为 Agent 架构。

## 十、常见问题

**Q1：必须使用 GPU 吗？**  
A：不需要。因为 Embedding 和 LLM 推理都通过云端 API 完成，后端只需要普通 CPU 服务器。

**Q2：Embedding API 与 DeepSeek API 可以不同厂商吗？**  
A：可以。例如 Embedding 使用阿里云，DeepSeek 使用硅基流动，只要在代码中分别配置即可。

**Q3：知识库更新频率如何？**  
A：建议文档发生变化时手动点击“更新知识库”按钮。也可以设置定时任务（如每天凌晨）自动调用 /rebuild 接口。

**Q4：系统能支持多少人同时使用？**  
A：单机后端可支持 20-50 人并发（取决于服务器配置和网络带宽）。如需更高并发，可水平扩展后端实例。

**Q5：如何保证回答的一定来自知识库？**  
A：通过三重机制：相似度阈值过滤、强约束提示词、低 temperature 参数。实测可将越界率控制在 1% 以下。
