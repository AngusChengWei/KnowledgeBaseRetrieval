# 任务：动态角色与菜单管理系统

## 任务概要

**目标**：把现有的"硬编码角色 + 硬编码菜单"系统，改造为"管理员可在后台动态配置角色和菜单"的灵活系统。

**预计工期**：10-12 个工作日（约 2-2.5 周）

**难度**：⭐⭐⭐（中等，适合实习中后期）

**前置技能**：
- 能看懂 Python 函数和 SQL（不需要精通）
- 会用 Vue 3 + Element Plus（看现有页面照葫芦画瓢即可）
- 理解 HTTP 请求/响应基本概念

## 一、现状理解（必读，动手前先看）

### 1.1 当前系统是怎么做角色和菜单的？

**角色**：写死在数据库 `users.role` 字段，只有 3 种值
- `super_admin`（超级管理员）
- `admin`（组织管理员）
- `user`（普通用户）

**菜单**：写死在 [Code/backend/auth.py:329-369](../Code/backend/auth.py#L329-L369) 的 `get_menus_for_role()` 函数

```python
def get_menus_for_role(role: str) -> list:
    base_menus = [
        {"key": "smart-qa", "label": "智能问答", "icon": "chat"},
        ...
    ]
    if role == 'super_admin':
        return base_menus + analysis_menus + knowledge_menus + admin_menus + super_admin_menus
    elif role == 'admin':
        return base_menus + analysis_menus + knowledge_menus + admin_menus
    else:
        return base_menus + analysis_menus
```

**前端使用**：[Layout.vue](../Code/ai-assistant/src/views/Layout.vue) 读取 `authStore.menus` 渲染侧边栏

### 1.2 当前系统的问题

1. **新增菜单要改 Python 代码**：哪怕只是加一个菜单项也要让开发改代码、重新部署
2. **角色固定死了 3 种**：客户想要"HR经理"、"财务主管"、"只读用户"等自定义角色完全做不到
3. **菜单和角色绑死**：想给某个 admin 临时禁用一个菜单做不到
4. **不支持组织定制**：不同公司可能想要不同的菜单和角色

### 1.3 改造后的效果

- 超级管理员可以在后台"角色管理"页面创建自定义角色（如"HR经理"）
- 超级管理员可以在"菜单管理"页面新增/修改/删除菜单项
- 给角色勾选哪些菜单可见
- 给用户分配角色（一个用户可以有多个角色）
- 用户登录后，系统自动根据其角色聚合出可见菜单

## 二、数据库设计

### 2.1 新增 4 张表

```sql
-- 1. 菜单表（替代原来硬编码的菜单数据）
CREATE TABLE menus (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    menu_key        TEXT UNIQUE NOT NULL,        -- 唯一标识，如 'smart-qa'
    label           TEXT NOT NULL,                -- 显示名称，如 '智能问答'
    icon            TEXT,                          -- 图标 key，如 'chat'
    route           TEXT NOT NULL,                -- 前端路由，如 '/smart-qa'
    group_name      TEXT,                          -- 分组名，如 'AI助手'
    sort_order      INTEGER DEFAULT 0,            -- 排序权重
    is_active       INTEGER NOT NULL DEFAULT 1,   -- 是否启用
    created_at      TIMESTAMP DEFAULT (datetime('now','localtime')),
    updated_at      TIMESTAMP DEFAULT (datetime('now','localtime'))
);

-- 2. 角色表
CREATE TABLE roles (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    code            TEXT NOT NULL,                -- 角色编码，如 'hr_manager'
    name            TEXT NOT NULL,                -- 角色名称，如 'HR经理'
    description     TEXT DEFAULT '',
    is_system       INTEGER NOT NULL DEFAULT 0,   -- 1=系统内置（不可删除），0=自定义
    org_id          INTEGER REFERENCES organizations(id),  -- NULL=全局角色，非NULL=组织专属
    created_at      TIMESTAMP DEFAULT (datetime('now','localtime')),
    UNIQUE(code, org_id)                          -- 同一组织内角色编码唯一
);

-- 3. 角色-菜单关联表（哪个角色能看哪些菜单）
CREATE TABLE role_menus (
    role_id         INTEGER NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
    menu_id         INTEGER NOT NULL REFERENCES menus(id) ON DELETE CASCADE,
    PRIMARY KEY (role_id, menu_id)
);

-- 4. 用户-角色关联表（一个用户可有多个角色）
CREATE TABLE user_roles (
    user_id         INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    role_id         INTEGER NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
    PRIMARY KEY (user_id, role_id)
);
```

### 3.3 后端 API 设计

#### 角色管理 API（`role_router`）

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| `GET` | `/admin/roles` | admin+ | 列出当前组织所有角色（含系统角色） |
| `POST` | `/admin/roles` | super_admin | 创建自定义角色 |
| `PUT` | `/admin/roles/{role_id}` | super_admin | 修改角色（系统角色只能改名称，不能改code） |
| `DELETE` | `/admin/roles/{role_id}` | super_admin | 删除角色（系统角色不可删；已分配给用户的角色不可删） |
| `PUT` | `/admin/roles/{role_id}/menus` | super_admin | 配置角色的菜单（传 menu_id 数组） |
| `GET` | `/admin/roles/{role_id}/menus` | admin+ | 查询角色已分配的菜单 |
| `PUT` | `/admin/users/{user_id}/roles` | admin | 给用户分配角色（传 role_id 数组） |

#### 菜单管理 API（`menu_router`）

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| `GET` | `/admin/menus` | admin+ | 列出所有菜单 |
| `POST` | `/admin/menus` | super_admin | 创建菜单 |
| `PUT` | `/admin/menus/{menu_id}` | super_admin | 修改菜单 |
| `DELETE` | `/admin/menus/{menu_id}` | super_admin | 删除菜单（已被角色引用时禁止删除，应先解除引用） |

5. ✅ **修改菜单立即生效**：超管新增一个菜单 [报销系统]，重新登录后能看到（如果有权限）
6. ✅ **删除保护**：尝试删除系统角色（如 super_admin）→ 报错；尝试删除已被使用的角色 → 报错
7. ✅ **多角色聚合**：给用户分配两个角色 → 登录后菜单是两个角色菜单的并集
8. ✅ **向后兼容**：原有的 admin 用户登录后菜单和改造前完全一致

3. **本地跑起来**：能登录、能切换角色、能看到菜单变化

### 7.2 推荐的开发顺序

```
后端 → 前端 → 联调
具体某个功能 → 先后端 API + Postman 测通 → 再写前端 → 联调
```

不要一口气把所有后端都写完再写前端，会出现"前端发现后端设计不合理需要返工"的情况。

### 7.3 常见坑点提醒

| 坑 | 应对 |
|----|------|
| SQLite 的 `ALTER TABLE` 限制多 | 新建表用 `CREATE TABLE IF NOT EXISTS`，别想着 ALTER |
| 现有 `users.role` 字段不要删 | 保留它作为"主角色"，避免破坏现有 `require_admin` 等逻辑 |
| 菜单聚合查询要去重 | 用户多个角色可能配置了相同菜单，要用 `DISTINCT` 或 set |
| 前端图标动态加载 | Element Plus 图标需要按需 import，新菜单的 icon 要在 iconMap 中注册 |
| 测试时清空数据库 | 改完表结构记得删 `data/app.db` 重新初始化，否则旧表没有新字段 |
| 权限检查别忘了 | 角色管理 API 要校验权限：列表 admin+，写操作 super_admin |

### 7.4 不要做的事（避免画蛇添足）

- ❌ 不要顺手"优化" `auth.py` 现有代码——只新增，不重构
- ❌ 不要引入新的依赖库（项目里有什么用什么）
- ❌ 不要做"权限点（permission）"级别的精细控制——超出范围
- ❌ 不要做"菜单多级嵌套"——超出范围
- ❌ 不要在第一版就加缓存——先跑通功能，性能问题等真出现了再优化
- ❌ 不要写很长的 docstring——参考现有代码的注释密度

### 7.5 卡住时怎么办

1. **看现有代码怎么写的**：项目里 90% 的问题都能在现有代码中找到答案
2. **看类似的 CRUD 实现**：参考 [routes/admin.py](../Code/backend/routes/admin.py) 的部门管理（CRUD 模板）
3. **看前端类似页面**：参考 [Departments.vue](../Code/ai-assistant/src/views/admin/Departments.vue)（CRUD 模板）
4. **每天主动找带教 review 一次**：哪怕只完成一小部分
