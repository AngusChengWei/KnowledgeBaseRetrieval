# 企业AI知识助手系统 — 项目功能报告（含详细逻辑）

## 一、项目概览

| 维度 | 内容 |
|------|------|
| **项目名称** | KnowledgeBaseRetrieval（企业AI知识助手） |
| **架构** | 前后端分离：FastAPI 后端 + Vue 3 前端 |
| **部署** | Nginx 反向代理 + systemd 进程守护 |

| **开发语言** | Python 3.13+ / JavaScript (Vue 3) |

## 二、后端核心模块

### 1. 认证与授权 (`auth.py`)

#### 1.1 注册 (`register_user`)

**详细逻辑流程：**

```
① 校验用户名格式（正则 ^[a-zA-Z0-9_\u4e00-\u9fff]{3,32}$）和密码长度（≥6）
② 检查用户名唯一性（SELECT id FROM users WHERE username = ?）
③ 邀请码校验（必填，不可为空）：
   ├─ SELECT id, admin_invite_code, user_invite_code FROM organizations
   │  WHERE admin_invite_code = ? OR user_invite_code = ?
   ├─ 无匹配 → 400 "邀请码无效"
   └─ 匹配 → 获取 org_id
④ 检查所属组织是否被禁用：_check_org_active(db, org_id)
⑤ 确定角色：
   ├─ 邀请码匹配 admin_invite_code → role = 'admin'
   └─ 匹配 user_invite_code → role = 'user'
⑥ 密码哈希：PBKDF2-SHA256（600,000轮迭代 + 随机32字节hex盐）
⑦ INSERT INTO users (username, password_hash, salt, role, org_id)
⑧ 返回 {"user_id", "username"}
```

#### 1.2 登录 (`login_user`)

**详细逻辑流程：**

```
① 校验用户名格式和密码长度
② 查询用户：SELECT id, password_hash, salt, role, status, org_id FROM users
③ 验证密码：pbkdf2_hmac("sha256") 比对 hash
   ├─ 不匹配 → 401 "用户名或密码错误"
   └─ 匹配 → 继续
④ 检查用户状态：
   ├─ status = 'disabled' → 403 "账号已被禁用"
   └─ active → 继续
⑤ 检查组织状态：_check_org_active(db, user["org_id"])
   ├─ 组织 disabled → 403 "所属组织已被禁用"
   └─ active → 继续
⑥ 删除该用户旧的 token：DELETE FROM auth_tokens WHERE user_id = ?
⑦ 生成新 token：secrets.token_urlsafe(32)（24小时有效期）
⑧ INSERT INTO auth_tokens (token, user_id, created_at, expires_at)
⑨ 查询用户部门：get_user_departments()
⑩ 查询组织编码：SELECT code FROM organizations WHERE id = ?
⑪ 返回 {token, user_id, username, role, org_id, org_code, departments}
```

#### 1.3 Token 验证 (`get_current_user`)

**详细逻辑流程：**

```
① 从 Authorization header 提取 token（格式 "Bearer <token>"）
② 联合查询 auth_tokens + users：
   SELECT u.id, u.username, u.role, u.status, u.org_id, t.expires_at
   FROM users u JOIN auth_tokens t ON u.id = t.user_id
   WHERE t.token = ?
③ token 不存在 → 401 "无效的登录凭证"
④ 检查 token 是否过期（expires_at vs datetime.now()）：
   ├─ 过期 → 清除过期 token → 401 "登录已过期"
   └─ 有效 → 继续
⑤ 检查用户状态 → disabled 则 403
⑥ 检查组织状态 → _check_org_active(db, row["org_id"])
   ├─ disabled → 403 "所属组织已被禁用"
   └─ active → 继续
⑦ 查询部门 + 组织编码
⑧ 返回 {user_id, username, role, org_id, org_code, departments}
```

#### 1.4 _check_org_active 辅助函数

```
参数: db, org_id（可为 None）
① org_id 为 None → 直接返回
② SELECT status FROM organizations WHERE id = ?
③ status == 'disabled' → raise HTTPException(403)
```

#### 1.5 角色权限体系

```
┌─────────────────────────────────────────────────┐
│                     super_admin                    │
│  ┌───────────── 拥有所有权限 ─────────────┐      │
│  │ 组织管理(跨组织CRUD) / 用户管理 /        │      │
│  │ 部门管理 / 知识库 / AI助手 / 文件分析    │      │
│  └─────────────────────────────────────────┘      │
│         可通过 header X-Current-Org-Id 切换组织     │
├─────────────────────────────────────────────────┤
│                  admin (组织管理员)                 │
│  ┌────── 限本组织操作 ──────┐                     │
│  │ 用户管理 / 部门管理 /     │                     │
│  │ 知识库 / 公司设置 /       │                     │
│  │ AI助手 / 文件分析          │                     │
│  └──────────────────────────┘                     │
├─────────────────────────────────────────────────┤
│                  user (普通用户)                    │
```

### 2. 多租户组织管理

#### 2.1 有效组织 ID 解析 (`_get_effective_org_id`)

**详细逻辑：**

```
参数: user（当前用户）, request（HTTP请求）
① 若 role == 'super_admin'：
   ├─ 检查 header X-Current-Org-Id
   ├─ 有值 → 返回 int(header_org)（超管切换到指定组织）
   └─ 无值 → 返回 None（超管查看全部）
② 若 role != 'super_admin'：
   └─ 返回 user.get("org_id")（绑定自己的组织）
```

#### 2.2 组织上下文获取 (`_get_effective_org_context`)

```
参数: user, request
① 调用 _get_effective_org_id 获取 org_id
② 若超管切换了组织（org_id != user.org_id）：
   └─ 查询 SELECT code FROM organizations WHERE id = ? 获取 org_code
③ 返回 (effective_org_id, org_code)
```

#### 2.3 数据隔离规则

| 场景 | 隔离逻辑 |
|------|----------|
| **用户查询列表** | WHERE org_id = ?（admin+ 只能看本组织） |
| **超管查看所有** | org_id = None（不传则不加 WHERE 条件） |
| **超管切换后查看** | WHERE org_id = ?（切换到指定组织） |
| **知识库文档路径** | `knowledge_docs/{org_code}/{dept}/`（按组织编码隔离目录） |
| **Chroma 集合名** | `kb_{org_code}_{dept}`（按组织编码隔离向量库） |
| **会话创建** | INSERT 时写入 org_id |
| **会话查询** | WHERE org_id = ?（按组织隔离） |
| **管理接口** | 每个接口调用 `_get_effective_org_id()` 动态获取 |

#### 2.4 组织禁用拦截

**三处拦截点（auth.py）：**

```
┌─────────────────────────────────────────────┐
│              组织状态实时检查                    │
│                                              │
│  登录 → register_user() → _check_org_active │
│  登录 → login_user()    → _check_org_active │
│  每次API请求 → get_current_user() → _check_org_active │
│                                              │
│  任意一点触发 disabled → HTTP 403 + 提示信息   │
│  "所属组织已被禁用，请联系系统管理员"            │
└─────────────────────────────────────────────┘
```

### 3. RAG 问答引擎

#### 3.1 问答接口完整流程 (`POST /ask`)

**详细逻辑（含源代码级别的 10 步流程）：**

```
┌──────────────────────────────────────────────────────────────┐
│  POST /ask 完整处理流程                                       │
├──────────────────────────────────────────────────────────────┤
│  输入: {session_id, question, department}                     │
├──────────────────────────────────────────────────────────────┤
│  ① 组织上下文解析                                            │
│     _get_effective_org_context(user, request) → (org_id, code)│
│                                                              │
│  ② 部门合法性校验                                            │
│     _validate_department(department, effective_org_id)        │
│     SELECT id FROM departments WHERE name=? AND org_id=?      │
│                                                              │
│  ③ 权限校验                                                  │
│     get_user_accessible_departments(user) → 用户可访问部门列表 │
│     if department not in accessible → 403                     │
│                                                              │
│  ④ 会话所有权校验                                             │
│     get_session_owner(session_id) → owner_id                  │
│     if owner_id != user_id → 403 "无权访问"                    │
│                                                              │
│  ⑤ 更新会话部门（用户可能切换了知识库）                         │
│     update_session_department(db, session_id, new_dept)       │
│     → UPDATE department + 追加到 departments JSON 列表        │
│                                                              │
│  ⑥ 保存用户消息到数据库                                       │
│     add_message(db, session_id, "user", question)             │
│                                                              │
│  ⑦ Query Rewriting（LLM 改写查询）                            │
│     rewrite_query(question) → rewritten                       │
│     ├─ 系统提示词：提取核心关键词、补充同义词、去除口语化       │
│     ├─ temperature=0.3, max_tokens=100                        │
│     ├─ 成功 → 使用改写后文本                                  │
│     └─ 失败(Exception) → 回退原始问题                          │
│                                                              │
│  ⑧ Embedding 向量化                                          │
│     get_embedding(rewritten) → 向量（DashScope API）          │
│                                                              │
│  ⑨ Chroma 向量检索                                           │
│     search_similar(embedding, top_k=5,                       │
│                    department=dept, org_code=org_code)        │
│     ├─ 集合名: kb_{org_code}_{dept}                          │
│     ├─ 检索返回: {text, metadata, similarity}                 │
│     ├─ similarity = 1 - distance（cosine 距离转相似度）       │
│     └─ 阈值过滤: similarity >= SIMILARITY_THRESHOLD(0.3)     │
│                                                              │
│  ⑩ LLM 生成回答                                              │
│     ├─ 有检索结果(>=1条) → 组装参考资料文本 + 问题            │
│     │  → generate_answer(question, context_chunks, history)  │
│     └─ 无检索结果 → 调用 LLM 以助手身份友好回复               │
│        → generate_answer(question, [], history)              │
│        （问候则友好回应，业务问题则提示资料暂缺）              │
│                                                              │
│  ⑪ 保存 AI 回复（含 sources JSON）                           │
│     add_message(db, session_id, "assistant", answer, sources) │
│                                                              │
│  ⑫ 记录审计日志                                             │
│     log_action("ask_question", detail="部门=..., 问题=...")   │
│                                                              │
│  ⑬ 返回 {answer, sources[], session_id}                     │
└──────────────────────────────────────────────────────────────┘
```

#### 3.2 Query Rewriting (`llm.py`)

**详细逻辑：**

```
输入: 用户原始问题 "咱们公司年假是怎么算的？"
↓
System Prompt: "提取核心关键词和概念，补充同义词，去除口语化，
                输出不超过50字的检索查询"
↓
DeepSeek API 调用: temperature=0.3, max_tokens=100
↓
输出: "公司年假政策 年假计算规则"
↓
失败则回退到原始问题
```

#### 3.3 LLM 提示词策略 (`llm.py`)

**有检索结果时的消息组装逻辑：**

```
messages = [
  {"role": "system", "content": "你是一个企业内部知识库问答助手..."},
  ...history[-20条]...,  # 保留最近10轮对话
  {"role": "user", "content": "参考资料：\n{context_text}\n\n---\n\n用户问题：{question}\n\n请根据以上参考资料回答用户的问题。"}
]
```

**无检索结果时的消息组装逻辑：**

```
messages = [
  {"role": "system", ...},
  ...history[-20条]...,
  {"role": "user", "content": "用户问题：{question}\n\n知识库中没有找到该问题的相关资料。请以友好的企业知识助手身份回复用户：
  - 如果是问候/聊天，友好回应并简要介绍自己的功能
  - 如果是业务问题，说明知识库中暂无相关资料
  - 不要编造任何信息"}
]
```

**系统提示词核心规则（7条）：**
1. **仅根据参考资料回答**，不使用外部知识
2. 无相关信息时明确告知"根据现有资料无法回答该问题"
3. **禁止编造或猜测**
4. 回答要简洁、准确、有条理
5. 引用具体内容需标注**来源文件名**
6. **需用户补充信息时**：先展示相关政策标准，再主动追问所缺信息（例如先展示年假分段标准，再问工作年限）
7. 用户在后续对话中补充信息后，结合资料给出明确答案

#### 3.4 Embedding (`embedding.py`)

**详细逻辑：**

```
get_embedding(text):
  → 单条文本调用 DashScope TextEmbedding API
  → 返回向量 List[float]

get_embeddings_batch(texts, batch_size=10):
  → 分批处理（DashScope 单次最多 10 条）
  → 每批异步调用 _call_embedding_api
  → 合并所有向量返回

_call_embedding_api(texts):
  → TextEmbedding.call(model=config.EMBEDDING_MODEL, input=texts)
  → 校验 status_code == 200
  → 按 text_index 排序保证顺序一致
  → 返回 [item["embedding"] for item in embeddings_data]
```

#### 3.5 向量存储与检索 (`vector_store.py`)

**集合命名：**

```python
def _get_collection_name(department, org_code="default"):
    return f"knowledge_base_{org_code}_{department}"
```

**空间度量：** `{"hnsw:space": "cosine"}`（余弦距离）

**向量库重建 (`rebuild_collection`)：**

```
① 删除旧集合：client.delete_collection(collection_name)
② 创建新集合：client.create_collection(name, metadata={hnsw:space: "cosine"})
③ 生成唯一 IDs：["chunk_0", "chunk_1", ...]
④ 分批添加（每批 5000 条）：collection.add(ids, documents, embeddings, metadatas)
```

**多部门检索 (`search_similar_multi`)：**

```
输入: query_embedding, top_k_per_dept=3, departments=["general","hr"], org_code
↓
① 遍历每个部门：
   ├─ search_similar(query_embedding, top_k, dept, org_code)
   ├─ 标记每个结果来源 department
   └─ 以 text[:100] 为 key 去重（保留相似度最高的版本）
↓
② 按 similarity 降序排列
↓
输出: [{text, metadata, similarity, department}, ...]
```

**相似度计算：** Chroma 返回余弦距离 `distance`，转换为相似度 `similarity = 1 - distance`

#### 4.4 重建向量库 (`POST /rebuild`)

**完整处理流水线：**

```
输入: department
权限: require_admin
↓
① 解析组织上下文
② 部门校验
③ load_documents(docs_dir)：
   ├─ 遍历目录下所有支持格式文件
   ├─ 根据扩展名选择解析器（PDF/DOCX/TXT/XLSX/MD）
   └─ 返回 [{"filename", "content"}, ...]
④ split_documents(documents, chunk_size=500, overlap=50)：
   ├─ MD文件 → _split_markdown_by_headers（按标题语义分块）
   ├─ 其他文件 → _split_text_to_dicts（固定大小分块）
   └─ 返回 [{"text", "metadata": {"filename", "chunk_index", "section"}}, ...]
⑤ get_embeddings_batch(texts, batch_size=10) → 批量向量化
⑥ rebuild_collection(texts, embeddings, metadatas, dept, org_code)
   └─ 删除旧集合 → 创建新集合 → 分批添加(每批5000条)
⑦ 记录审计日志
⑧ 返回 {status, doc_count, chunk_count}
```

#### 4.5 文本分块详细算法 (`text_splitter.py`)

**Markdown 按标题分块 (`_split_markdown_by_headers`)：**

```
输入: md_content, max_chunk_size=500
↓
① 正则匹配标题行：r'^(#{1,4})\s+(.+)$'
② 按标题位置切割文档，保留层级标题栈（current_headers）
③ 每个段落附带章节上下文（"一级 > 二级 > 三级"）
④ 段落内容 prefix 格式: "[一级 > 二级] 段落内容"
⑤ 超长段落（>500字符）二次切分：
   ├─ 优先按换行分割
   └─ 单段仍超长 → 按 (max_size-50) 强制截断
↓
输出: [{"text": "...", "section": "..."}, ...]
```

**非 Markdown 固定分块 (`_split_text_to_dicts`)：**

```
输入: text, chunk_size=500, overlap=50
↓
while start < text_length:
  end = start + chunk_size
  chunk = text[start:end]
  if chunk.strip(): chunks.append(chunk)
  start = start + chunk_size - overlap
↓
输出: ["块0", "块1", ...]
```

#### 4.6 文档解析算法 (`document_loader.py`)

| 文件类型 | 解析方法 |
|----------|----------|
| **PDF** | PyPDF2.PdfReader → 逐页 extract_text() → 拼接 |
| **DOCX** | python-docx → 逐 paragraph 提取非空段落 |
| **TXT/MD** | chardet 自动检测编码 → decode → 支持 UTF-8/GBK/GB18030 |
| **XLSX** | openpyxl（read_only + data_only）→ 逐 sheet → 逐行 → " | " 连接单元格值 |

```python
def _clean_text(text: str) -> str:
    # 1. 替换排版字符
    text = text.replace("\u2014", "--")   # em dash →
    text = text.replace("\u2013", "-")    # en dash → -
    text = text.replace("\u2018", "'")    # 左单引
    text = text.replace("\u2019", "'")    # 右单引
    text = text.replace("\u201c", '"')    # 左双引
    text = text.replace("\u201d", '"')    # 右双引
    text = text.replace("\u2026", "...")  # 省略号
    text = text.replace("\u00a0", " ")    # 不换行空格

    # 2. 移除控制字符（保留换行符 \n）
    text = "".join(c for c in text if c == "\n" or (ord(c) >= 32 and ord(c) != 127))

    # 3. 移除 emoji 和装饰符号
    text = "".join(c for c in text if not _is_unsupported_char(c))
```

**_is_unsupported_char 过滤范围：**

```
U+2980-U+29FF  杂项数学符号-B
U+1F000-U+1FFFF Emoji 区块
U+2300-U+23FF  杂项技术符号
U+25A0-U+25FF  几何图形
U+2B00-U+2BFF  杂项符号和箭头
U+FE00-U+FE0F  变体选择符（emoji 修饰符）
```

## 五、安全机制详解

| 安全措施 | 实现细节 |
|----------|----------|
| **密码安全** | PBKDF2-SHA256（600,000轮迭代）+ 随机32字节hex盐；无明文存储 |
| **Token 认证** | `secrets.token_urlsafe(32)` 生成 43 字符 token；24h过期；过期自动清理 |
| **角色权限** | 后端 FastAPI Depends 装饰器 + 前端路由守卫 meta 字段双重校验 |
| **组织隔离** | `_get_effective_org_id()` 函数统一处理；所有查询 WHERE org_id = ? |
| **组织禁用** | 注册/登录/每次请求三处 _check_org_active() 拦截 |
| **用户禁用** | 登录时 CHECK status + get_current_user() 每次校验 |
| **超管保护** | 非超管不能查看/操作超管用户；修改用户角色仅限超管 |
| **邀请注册** | 邀请码匹配 organizations 表；匹配失败返回通用错误"邀请码无效" |
| **输入校验** | 用户名正则 + 密码最小长度；所有查询使用参数化绑定防 SQL注入 |
| **审计追踪** | try-except 包裹，失败不阻塞业务；detail 截断到500字 |

```
登录流程:
┌─────────┐    ┌────────────┐    ┌──────────────┐    ┌──────────────┐
```

### 8.2 RAG 问答流程

```
用户提问 "年假怎么算"
       │
       ▼
⑦ Query Rewriting ──────────→ 改写为 "公司年假制度 年假计算规则"
  (DeepSeek, temp=0.3)
       │
       ▼
⑧ Embedding 向量化
  (DashScope text-embedding-v3)
       │
       ▼
⑨ Chroma 向量检索 (top_k=5, threshold=0.3)
  ├─ search_similar(向量, department="hr", org_code="CompanyB")
  └─ 过滤 similarity >= 0.3
       │
       ├── 有结果 ──→ ⑩ AI 生回答（含参考资料）
       │                ├─ 消息组装: system + history + 参考文本 + 问题
       │                └─ DeepSeek, temp=0.1, max_tokens=2048
       │
       └── 无结果 ──→ ⑩ AI 友好回复
                        ├─ 问候 → 打招呼+自我介绍
                        └─ 业务问题 → "暂无相关资料"
       │
       ▼
⑪ 保存消息到数据库 (user + assistant + sources JSON)
⑫ 返回 {answer, sources[{filename, chunk}], session_id}
```

### 8.3 知识库文档处理流水线

```
上传文档 (POST /upload)
  ├─ 校验格式 (.pdf/.docx/.txt/.md/.xlsx)
  ├─ 校验大小 (≤10MB)
  └─ 写入 knowledge_docs/{org_code}/{dept}/
       │
重建向量库 (POST /rebuild)
  ├─ load_documents(docs_dir)
  │    ├─ .pdf → PyPDF2 逐页提取
  │    ├─ .docx → python-docx 逐段提取
  │    ├─ .txt/.md → chardet 编码检测
  │    └─ .xlsx → openpyxl 逐 sheet 逐行
  │
  ├─ split_documents(documents)
  │    ├─ .md → 按标题语义分块（标题栈 + 章节上下文）
  │    └─ 其他 → 固定大小分块 (500字符 + 50重叠)
  │
  ├─ get_embeddings_batch(texts, batch_size=10)
  │    └─ DashScope API 分批向量化
  │
  └─ rebuild_collection(texts, embeddings, metadatas)
       ├─ 删除旧 Chroma 集合
       ├─ 创建新集合 (hnsw:space="cosine")
       └─ 分批添加 (每批5000条)
```

### 8.4 数据文件校验流程

```
上传 Excel/CSV → analyze_data_file()
  │
  ├─ 读取文件
  │    ├─ .csv → UTF-8 → GBK → GB18030 三级编码尝试
  │    └─ .xlsx → pandas.read_excel(engine="openpyxl")
  │
  ├─ 概览统计
  │    ├─ 行数 / 列数 / 列名 / 数据类型 / 内存占用
  │    └─ 空文件 → 返回 error
  │
  ├─ 列分析（逐列）
  │    ├─ 非空/空 统计
  │    ├─ 数值列 → min/max/mean/std/unique/异常值(3σ法)
  │    └─ 文本列 → unique/空字符串/长度范围/Top5分布
  │
  ├─ 缺失值总览
  │    └─ 有缺失列 / 总缺失数 / 缺失占比
  │
  ├─ 重复行检测
  │    └─ df.duplicated().sum()
  │
  ├─ 异常值汇总（3σ 法）
  │    └─ 按列统计超出 mean±3std 的异常值
  │
  └─ AI 报告生成（可选关联知识库）
       ├─ 用文件摘要向量检索知识库
       └─ DeepSeek → 自然语言分析报告
```

### 8.5 合规检查流程

```
上传文档 → analyze_document_compliance()
  │
  ├─ 提取文档内容 (document_loader._parse_file)
  │    └─ 截取前8000字符
  │
  ├─ 第一次知识库检索
  │    ├─ doc_text[:2000] → Embedding
  │    ├─ search_similar_multi(top_k=3, departments, org_code)
  │    └─ 过滤 similarity >= max(0.3, 0.5)
  │         └── 无结果 → 第二次检索（关键词提取）
  │
  ├─ 构建 policy_context
  │    ├─ 有结果 → 拼接 [来源] (相似度) + 制度文本
  │    └─ 无结果 → "知识库无相关制度"
  │
  ├─ LLM 合规分析 (DeepSeek)
  │    ├─ 符合项（列出符合制度的条款）
  │    ├─ 不符合项（引用制度原文对比）
  │    ├─ 风险评估（高/中/低）
  │    ├─ 总体评分（优/良/中/差）
  │    └─ 改进建议（具体修改方向）
  │
  └─ 返回 {document_summary, compliance_report, related_policies[]}
```
